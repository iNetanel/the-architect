# How to Read SUCCESS.md

> Written after every successful The Architect run. Lives in the project root.

---

## What is SUCCESS.md?

After The Architect finishes all tasks, it writes a `SUCCESS.md` file to the root of your project. It is a plain-text run report — a snapshot of what happened: how long it took, which tasks ran, how many attempts each took, which models were used, whether the retrospective reviewer found issues, and how many tokens the AI consumed.

It is generated by `the_architect/core/success.py`.

## Section by Section

### Header

| Field | Meaning |
|-------|---------|
| `Date` | Timestamp when the run finished |
| `Duration` | Total wall-clock time (M:SS or H:MM:SS) |
| `Result` | `✓ All tasks completed` or `✗ N task(s) failed` |

---

### Tasks Table

| Column | Meaning |
|--------|---------|
| `Task` | Task ID (T01, T02 …, or R01, R02 … for retrospective fix-up tasks) |
| `Title` | The task description from the task file |
| `Status` | `✓ Done`, `✗ Failed`, or `○ Skipped` |
| `Attempts` | How many tries the task took (1 = first attempt success, 3 = two retries) |
| `Model` | Which AI model was used (provider prefix stripped for readability) |
| `Duration` | Wall-clock time for that single task |
| `Tokens` | **Total tokens for that task** = input + output only (see below) |

> **Note:** When using Claude Code as the provider, token counts are not available (Claude Code outputs plain text with no structured token events). Token columns will show `0` for Claude Code runs.

**Reading attempts:** If `Attempts` is 1, the task passed first try. If it's 3, the task failed twice and succeeded on the third attempt. In the terminal summary, tasks with >1 attempt are highlighted in yellow.

**Reading model:** Long provider prefixes like `anthropic/` or `openrouter/` are stripped for readability. The full model names appear in the Totals section.

---

### Totals — Token Fields Explained

This is the section that causes the most confusion. Here is what each number means:

#### `Total tokens: 50.8K`

This is **input + output only**. It is the "billable work" number — the tokens the model actually processed to think and respond.

```python
# From runner.py
@property
def total(self) -> int:
    return self.input_tokens + self.output_tokens
```

Cache tokens are **not included** in this total.

---

#### Token Breakdown

```
input 207 · output 50.6K · cache read 5210.4K · cache write 343.7K
```

| Field | What it is | Billed? |
|-------|------------|---------|
| `input` | Fresh tokens sent to the model as prompt (not from cache) | Yes — full price |
| `output` | Tokens the model generated as its response | Yes — full price |
| `cache read` | Tokens retrieved from Anthropic's prompt cache | Yes — at ~10% of normal price |
| `cache write` | Tokens written into the prompt cache for future reuse | Yes — at ~125% of normal price (one-time) |

---

#### Retries

```
Retries: 2 across 12 tasks
```

Total attempts minus total tasks. If 12 tasks took 14 total attempts, there were 2 retries. This tells you how much extra work was needed beyond the ideal first-try path.

---

#### Rate Limits Hit

```
Rate limits hit: 1 (T05)
```

Only shown when rate limits were detected. Lists which tasks were affected. In free mode, rate limits trigger automatic model rotation — the task still completes, just on a different model.

---

### Retrospective Section

Only appears when retrospective rounds were configured (`retrospective_rounds > 0`).

| Column | Meaning |
|--------|---------|
| `Round` | Which retrospective round (1-based) |
| `Issues Found` | Number of quality issues the reviewer identified |
| `Fix-up Tasks` | R-prefixed task IDs created to fix those issues, or `—` if none |
| `Duration` | Wall-clock time for the reviewer to run |

**Reading the retrospective:** If Round 2 shows `—` for fix-up tasks, the reviewer found no issues and the build is clean. If Round 1 created R01 and R02, those fix-up tasks appear in the Tasks table above with their own attempts, model, and token counts.

---

### Insights Section

| Metric | Meaning |
|--------|---------|
| `Avg duration per task` | Total duration ÷ number of done tasks |
| `Slowest task` | Task with the highest wall-clock time |
| `Avg tokens per task` | Total tokens ÷ number of done tasks |
| `Throughput` | Total tokens ÷ total minutes — how fast the AI was working |
| `Most tokens` | Task that consumed the most input + output tokens |
| `Most retries` | Task that required the most attempts (only shown when >1) |

## Why Total ≠ Cache Write (and why the numbers look strange)

Looking at the last run:

```
Total tokens:    50.8K
cache write:    343.7K
cache read:   5,210.4K
```

At first glance this looks wrong — how can cache read be 100× bigger than total?

**It makes complete sense.** Here is why:

### The AI has a long memory — and it is cached

Every time The Architect runs a task, it sends the AI a large context: the project files, the task instructions, the execution protocol, and the conversation history. This context can be **hundreds of thousands of tokens** long.

Anthropic's **prompt caching** means:
1. The first time a large prompt is sent, Anthropic stores it in a cache → those tokens are counted as **cache write**
2. Every subsequent API call that reuses that same prompt prefix retrieves it from cache → those tokens are counted as **cache read** (much cheaper)
3. Only the small "new" part of each prompt (the fresh question or instruction) counts as **input**

### The numbers from the last run, explained

| Metric | Value | Explanation |
|--------|-------|-------------|
| `input` | 207 tokens | The genuinely new text sent per call — tiny, because almost everything was cached |
| `output` | 50.6K tokens | All the text the AI wrote across 12 tasks — the actual work product |
| `cache write` | 343.7K tokens | The large shared context written to cache once (project files, prompts, protocol) |
| `cache read` | 5,210.4K tokens | That same cached context retrieved ~15× across many API calls during 12 tasks |
| **total** | **50.8K tokens** | input + output — the "thinking" work, excluding cache mechanics |

### Why is cache read so large?

Each task makes multiple API calls (planning, execution, verification steps). Each call retrieves the full cached context. With 12 tasks × multiple calls per task, the cache read accumulates fast — but you are paying only ~10% of the normal rate for those tokens.

**Cache read being large is a good sign.** It means the system is efficiently reusing context instead of re-sending it fresh every time.

---

## Why Total is the Right Number to Watch

`Total tokens` (input + output) is the number that tells you:

- **How much thinking the AI did** — how much novel reasoning and code generation happened
- **A fair basis for cost comparison** — because cache prices vary by provider and change over time
- **The "work" number** — the tokens that actually produced your code changes

Cache numbers are infrastructure overhead. They tell you about efficiency, not about how much the AI worked.

## Quick Reference — Token Types

```
┌─────────────────────────────────────────────────────────────────┐
│  WHAT YOU SEND TO THE AI                                        │
│                                                                 │
│  ┌──────────────────────────────┐  ┌──────────────────────┐    │
│  │  CACHED CONTEXT              │  │  FRESH INPUT         │    │
│  │  (project files, prompts,    │  │  (new instruction    │    │
│  │   history)                   │  │   or question)       │    │
│  │                              │  │                      │    │
│  │  → cache_write (first time)  │  │  → input_tokens      │    │
│  │  → cache_read  (reuse)       │  │                      │    │
│  └──────────────────────────────┘  └──────────────────────┘    │
│                                                                 │
│  WHAT THE AI SENDS BACK                                         │
│                                                                 │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │  GENERATED RESPONSE (code, analysis, decisions)          │  │
│  │  → output_tokens                                         │  │
│  └──────────────────────────────────────────────────────────┘  │
│                                                                 │
│  total = input_tokens + output_tokens  (cache excluded)        │
└─────────────────────────────────────────────────────────────────┘
```

## What SUCCESS.md Does NOT Show (Known Gaps)

The following information exists in the system but is **not yet surfaced** in SUCCESS.md:

| Missing info | Where the data exists | Why it matters |
|---|---|---|
| **Per-attempt details** | `run_task` loops over attempts, collapses to final `TaskResult` | Attempt 1 failed with a syntax error, attempt 2 succeeded → you only see "Done, 2 attempts" |
| **Circuit breaker events** | `on_circuit_event` callbacks in `run_all` | If the circuit opened, replanned, or cooldown-waited — invisible |
| **Tool calls / steps** | `step_start`/`step_finish` events parsed but discarded | No trace of how many file reads, edits, or bash commands the agent ran |

### Where to find the missing info today

| Info | Where to look |
|------|--------------|
| Per-attempt logs | `.architect/logs/T01.log`, `.architect/logs/T01.attempt2.log` |
| Retrospective logs | `.architect/logs/reviewer_round1.log` |
| Circuit breaker state | `.architect/circuit.json` |
| Full event log | The log files above contain raw provider output — JSON events (OpenCode) or plain text (Claude Code) |

## Source Files

| File | Role |
|------|------|
| `the_architect/core/success.py` | Writes SUCCESS.md and prints the terminal summary |
| `the_architect/core/runner.py` | `TokenUsage` model — defines and accumulates all four token fields; `stream_provider` — unified provider output engine |

| `the_architect/core/circuit.py` | Circuit breaker — tracks failure patterns, cooldowns, replans |
| `the_architect/cli.py` | Orchestrates planning → execution → retrospective loop |
